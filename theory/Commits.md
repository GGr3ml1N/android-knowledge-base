В профессиональной разработке стандартом де-факто является Conventional Commits. Это делает историю проекта понятной, позволяет автоматически генерировать списки изменений (Changelog) и упрощает поиск багов.

Вот основные правила «хорошего тона»:
## 1. Структура коммита
Хороший коммит состоит из заголовка, (опционально) тела и (опционально) футера.
```xml
<type>(<scope>): <subject>

[body]

[footer]
```
## 2. Типы коммитов (type)
Это самое важное. Ты сразу говоришь, что именно произошло:
- `feat`: Новая функциональность (например, добавил экран тренировки).
- `fix`: Исправление бага.
- `refactor`: Изменение кода, которое не меняет поведение (как наш переезд на аннотации Koin).
- `test`: Всё, что связано с тестами (наш случай сегодня).
- `docs`: Изменения в документации (README, KDoc).
- `chore`: Рутинные задачи (обновил версию библиотеки, поменял .gitignore).
- `style`: Правка форматирования (пробелы, отступы), не влияющая на код.

## 3. Правила заголовка (subject)
1. Используй повелительное наклонение: Пиши «Add test», а не «Added test» или «Adding test». Представь, что ты даешь команду коду: «Этот коммит должен исправить (fix) ошибку...».
2. Не ставь точку в конце.
3. Лимит — 50 символов. Заголовок должен быть кратким.
4. Первая буква — маленькая (в классическом стандарте) или большая (зависит от правил команды).

## 4. Тело коммита (body)
Если изменений много или они неочевидны, напиши через пустую строку от заголовка:
- Зачем это сделано? (самое важное).
- Кратко что изменилось, если файлов очень много.

### Пример «Плохо» vs «Хорошо»
**❌ Плохо:**
 - `Fixed some bugs` (какие? где?)
 - `Added tests` (какие тесты? чего?)
 - `Refactoring koin` (зачем? что сломалось?)

**✅ Хорошо (Conventional):**
 - `test(shared): add unit tests for workout mappers`
 - `refactor(di): migrate root component to koin annotations`
 - `fix(db): correct timestamp type in workout table`

Главный совет: `**Атомарность**`
Не смешивай всё в одну кучу. Если ты чинил баг и заодно решил отрефакторить половину проекта — сделай два разных коммита.
- Сначала `refactor: ...`
- Потом `fix: ...`

Это позволит тебе легко «откатить» (revert) только исправление бага, не теряя при этом рефакторинг, если что-то пойдет не так.
Для твоей текущей работы идеальным будет:

```
test(shared): add domain mappers and repository unit tests
refactor(koin): move root component to factory annotation for compile safety
```
(Можно сделать это одним коммитом, если изменения тесно связаны, но лучше двумя).
